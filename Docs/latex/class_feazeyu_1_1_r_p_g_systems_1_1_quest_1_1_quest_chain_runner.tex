\doxysubsection{Detailed Description}
Runtime driver for a Quest\+Kind.\+Chain quest graph. 

The chain is a static dependency description, not a linear flow:\+ every {\ttfamily Run\+Subgraph} node represents a quest; edges mean "{}prerequisite"{}. The runner maintains completion state and exposes the topological frontier as the set of Available quests.

── Supported quest kinds ──────────────────────────────────── Each \doxylink{namespace_feazeyu_1_1_r_p_g_systems_1_1_quest}{Quest} Reference node\textquotesingle{}s "{}\+Quest"{} field must be linked to a blackboard variable. The variable type decides how the quest is driven:\+ 
\begin{DoxyItemize}
\item Variable of type {\bfseries{Quest\+Graph}} holding a Quest\+Graph\+Asset:\+ Start\+Quest spawns a child Game\+Object with a Quest\+Runner that walks the quest\textquotesingle{}s graph, and the chain observes its {\ttfamily On\+Quest\+Ended} event. 
\item Variable of type {\bfseries{\doxylink{namespace_feazeyu_1_1_r_p_g_systems_1_1_quest}{Quest}}} holding a Quest\+Asset (simple quest, no graph):\+ Start\+Quest does not spawn a runner — it calls Quest\+Asset.\+Start on the asset and waits for external code to call Notify\+External\+Quest\+Completed (or Notify\+External\+Quest\+Failed) when the player\textquotesingle{}s condition is satisfied. 
\end{DoxyItemize}Supporting both lets chains mix graph-\/based quests (with objectives and rewards) and simple flag quests without forcing every quest to be a full graph.
